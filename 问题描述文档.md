## 📌 改造目标（用户需求）

1. **修复健康问题**：数据一致性/竞态条件导致的状态丢失
2. **并发能力保障**：支持100+会话同时进行（当前10-50）
3. **多worker部署**：支持Uvicorn 4-worker模式
4. **关键风险控制**：数据迁移零丢失 + 新系统稳定性验证

## 🏗️ 当前系统架构

```
用户请求（微信/Web）
    ↓
Cloudflare Tunnel (HTTPS)
    ↓
FastAPI + Uvicorn (单进程，reload模式)
    ├─ Agent Loop (7步)
    ├─ State Manager (JSON文件，data/sessions/{brand}/{user}.json)
    ├─ BM25 Knowledge Engine (进程内存，lazy load)
    └─ Claude SDK Client (文件系统session: ~/.claude/projects/.../*.jsonl)
    ↓
外部依赖
    ├─ Claude API (推理3-5s，主要瓶颈)
    ├─ 有赞API (订单查询200-500ms)
    └─ 链伴API (标签同步50-100ms)
```

**关键组件说明**：

1. **UserState存储**：JSON文件（~5KB/用户）
   - 路径：`data/sessions/{brand_id}/{user_id}.json`
   - 17个核心字段：user_id, brand_id, session_id, stage, sop_state, tags, milestones...
   - 保存机制：临时文件 + `os.replace()`原子替换

2. **Claude SDK Session**：文件系统（CLI管理）
   - 路径：`~/.claude/projects/{project}...{session-id}.jsonl`
   - 存储对话历史（system prompt + 所有消息）
   - Resume机制：传入session_id，CLI加载历史

3. **MCP工具系统**：10个工具（registered in wrapper.py L49-65）
   - `update_tags` → 修改state.tags → save()
   - `update_sop_state` → 修改completed_steps → save()
   - `generate_order_link` → 修改quoted=True → save()
   - 每个工具内部：`get_or_create()` → 修改 → `save()`（独立事务）

4. **SOP调度器**：基于rules.yaml的触发系统
   - Rules → Goals → Flows → Prompts（4层配置）
   - 触发时机：每次请求检查规则（时间条件+状态条件）
   - Wrapper初始化流程：`enter_flow()` → `enter_stage()` → `set_milestone()`

## 🔴 问题1: UserState并发写入竞态（数据丢失）

### 代码位置
- `backend/agent/state.py:604-631` (save方法)
- `backend/agent/wrapper.py:477-480` (重新加载state)
- `backend/tools/*.py` (所有工具的state修改)

### 问题描述

**场景A: 工具并发修改不同字段**
```
用户发送："我要买胶原蛋白"
  ↓
Agent调用工具：
  T1: update_tags() → 读state(v0) → tags.needs=["产品"] → save(v1)
  T2: generate_order_link() → 读state(v0) → quoted=True → save(v2覆盖v1)

结果：tags.needs丢失（因为v2基于v0，不包含v1的修改）
```

**场景B: 多个请求同时处理同一用户**
```
用户快速连发两条消息（间隔<1秒）：
  请求A: "我要买" → Worker1处理 → 修改quoted=True
  请求B: "我有过敏" → Worker2处理 → 修改tags.barriers=["过敏"]

如果Worker2读取的是旧state，quoted=True会丢失
```

**场景C: Wrapper重新加载state的竞态**
```
wrapper.py L477-480在SDK返回后重新加载state：
  state = await state_manager.get_or_create(user_id, brand_id)  # 重新读取
  state.session_id = msg.session_id
  await state_manager.save(state)

风险：如果其他worker在SDK调用期间修改了state，这次save会覆盖
```

### 关键代码片段

```python
# state.py L604-631: 当前保存逻辑
async def save(self, state: UserState) -> None:
    file_path = self._get_file_path(state.user_id, state.brand_id)

    # 更新时间戳
    state.last_interaction_at = datetime.now()

    # 原子文件写入（避免半写入损坏）
    tmp_path = file_path.with_suffix(file_path.suffix + ".tmp")
    with open(tmp_path, "w", encoding="utf-8") as f:
        json.dump(state.to_dict(), f, ensure_ascii=False, indent=2)  # ← 完整对象覆盖

    os.replace(tmp_path, file_path)  # ← 原子操作，但无法防止Last-Write-Wins
```

```python
# tools/tags.py L127-141: 典型工具的state修改模式
async def update_tags(args: dict[str, Any]) -> dict[str, Any]:
    user_id = args.get("user_id")

    # 重新获取state（未使用wrapper传入的state）
    state = await state_manager.get_or_create(user_id, get_current_brand_id())

    # 修改state字段
    state.update_tags(needs=needs, stage=stage, ...)
    state.tags_applied = True

    # 立即保存（完整对象覆盖）
    await state_manager.save(state)  # ← 无并发控制

    return {"content": [...], "is_error": False}
```

### 实测影响

- **单进程单用户**：5-10%概率（用户快速连发消息）
- **单进程多用户**：1-2%概率（不同用户并发）
- **多进程同用户**：30-50%概率（负载均衡轮询）
- **丢失内容**：tags追加、milestones更新、SOP步骤记录、quoted状态

### 根本原因分析

1. **无并发控制**：state.py无asyncio.Lock或文件锁
2. **完整对象覆盖**：`json.dump(state.to_dict())`写入所有字段，非增量更新
3. **工具独立事务**：每个工具内部`get→modify→save`，无全局事务保护

## 🔴 问题2: SOP调度器重复触发（业务逻辑错误）

### 代码位置
- `backend/agent/wrapper.py:140-224` (scheduler逻辑)
- `backend/config/brands/guanzhan/goals.yaml` (目标定义)
- `backend/config/brands/guanzhan/rules.yaml` (规则定义)

### 问题描述

**场景A: 多步流程触发后milestone未更新**
```
官栈Day1欢迎流程（9步）：
  T0: 用户创建，状态 {stage: new, welcome_done: false}
  T1: 规则匹配 → goal="welcome" → enter_flow("day1_welcome")
  T2: wrapper.py L206: scheduler_goal_mapping["welcome"] = []  ← 空数组！
  T3: 未设置milestone → welcome_done仍为false
  T4: 用户回复"收到" → 规则再次匹配 → 重复触发welcome流程
```

**场景B: 多worker并发检查规则**
```
用户在Day3，两个请求同时到达：
  Worker1: 检查Day3规则 → 匹配 → goal="taste_feedback"
  Worker2: 同时检查Day3规则 → 匹配 → goal="taste_feedback"
  Worker1: enter_flow("day3") → set_milestone(缺失)
  Worker2: enter_flow("day3") → 幂等检查通过，但milestone未更新
  结果：下次请求再次触发Day3流程
```

### 关键代码片段

```python
# wrapper.py L206-217: scheduler_goal_mapping定义
scheduler_goal_mapping = {
    "welcome": [],  # ← 多步流程，不立即标记milestone
    "taste_feedback": [],  # ← 问题！触发后状态未更新
    "exploring_started": [],
    "converted_followup": [],

    "long_term_followup_1": ["followup_14_done"],  # 单步流程才标记
    "coupon_20": ["coupon_20_sent"],
    "coupon_50_final": ["coupon_50_sent"],
}
```

```python
# wrapper.py L153-173: SOP初始化逻辑（缺少幂等检查）
if goal:
    sop_flow = get_sop_flow_for_goal(goal, brand_id, "default")
    if sop_flow:
        flow_id = f"{brand_id}_{sop_flow.id}"

        # ⚠️ 缺少goal级别幂等检查：
        # if state.get_milestone(f"{goal}_triggered"): return

        if state.current_flow != flow_id:  # flow级别幂等（不够）
            state.enter_flow(flow_id)
            state.enter_stage(first_stage_id)
            await state_manager.save(state)
```

### 实测影响

- **单进程**：10-20%概率（用户快速回复，Agent未标记milestone）
- **多进程**：40-60%概率（两个worker同时检查规则）
- **后果**：
  - 用户收到重复的Day1欢迎语（体验差）
  - 重复推送优惠券（浪费营销资源）
  - SOP完成率统计错误（重复计数）

### 根本原因分析

1. **scheduler_goal_mapping为空数组**：多步流程触发时不立即标记
2. **无goal级别幂等检查**：wrapper.py L153未检查`{goal}_triggered`字段
3. **依赖Agent调用update_sop_state**：但Agent可能在流程中断时未调用

## 🟡 问题3: BM25引擎冷启动竞态（性能浪费）

### 代码位置
- `backend/knowledge/bm25_engine.py:104-114` (_ensure_loaded方法)

### 问题描述

```
容器启动后，10个用户同时查询知识库：
  Worker1-10: _ensure_loaded("faq") → 未加载 → 开始加载JSONL + jieba分词
  结果：10个worker都加载一遍（每个150ms），浪费1.5秒CPU + 5MB内存×10
```

### 关键代码片段

```python
# bm25_engine.py L104-114
def _ensure_loaded(self, brand_id, line_id, collection_type):
    collection_name = f"{brand_id}_{line_id}_{collection_type}"

    if self.engine.get_collection(collection_name) is not None:
        return  # ← 已加载，直接返回

    # ⚠️ 问题：多个worker同时进入这里，重复加载
    jsonl_path = get_jsonl_path(brand_id, line_id, collection_type)
    self.engine.load_collection(collection_name, jsonl_path)  # 150ms jieba分词
```

### 实测影响

- **影响范围**：仅在容器重启后的前10-20个请求
- **资源浪费**：CPU时间1.5秒 + 内存50MB（临时）
- **用户体验**：首次查询延迟150ms（后续14ms）

## 🎯 改造需求总结

### 功能性需求（P0）

1. **修复UserState并发写入竞态**
   - 要求：0%数据丢失率（强一致性）
   - 场景：多worker同时处理同一用户，tags/milestones/SOP状态不丢失

2. **修复SOP调度器重复触发**
   - 要求：同一goal只触发一次（幂等性）
   - 场景：多worker同时检查规则，不重复发送欢迎语/优惠券

3. **支持多worker部署**
   - 要求：支持4-worker，120并发会话（4×30）
   - 约束：响应时间不降低（P95 < 5秒）

### 非功能性需求（P0）

4. **数据迁移安全**
   - 要求：零丢失 + 可回滚
   - 方案：全量备份 + 逐条验证 + 一键回滚脚本

5. **系统稳定性验证**
   - 要求：7天灰度期无P0故障
   - 方案：单元测试 + 集成测试 + 压力测试 + 灰度发布

### 性能需求（P1）

6. **优化BM25冷启动**
   - 要求：多worker不重复加载
   - 可选：引入缓存或分布式锁

## 📋 关键约束条件

### 技术约束

1. **Claude SDK Session机制**（已验证）
   - Session存储在文件系统（~/.claude/projects/.../*.jsonl）
   - 跨worker天然共享，无需特殊处理
   - Brand切换时必须清空session_id（wrapper.py L327-339已实现）

2. **MCP工具独立事务**（已验证）
   - 每个工具内部`get_or_create()` → 修改 → `save()`
   - 工具不接收state参数，必须重新获取

3. **SOP配置文件**（不可修改）
   - rules.yaml, goals.yaml, stage_flows.yaml, sop_prompts.yaml
   - 经过精心调试，不能随意修改内容
   - 只能修改触发逻辑（wrapper.py）

### 业务约束

4. **向后兼容**
   - 现有data/sessions/**/*.json数据必须无缝迁移
   - API接口不能破坏性变更

5. **降级策略**
   - 任何时刻可切回文件模式（环境变量）
   - 新系统故障时，不影响业务连续性

## 🤔 思考题（供开发者独立设计方案）

### 问题1: UserState并发控制

**现状**：完整对象覆盖 + 无锁 + 工具独立事务

**可能方案**：
- 方案A：应用层锁（asyncio.Lock或文件锁）
- 方案B：乐观锁（版本号+冲突重试）
- 方案C：数据库事务（Postgres/SQLite）
- 方案D：增量更新（PATCH而非PUT）
- 方案E：事件溯源（Event Sourcing）

**请思考**：
1. 哪种方案最适合当前架构（考虑工具独立事务的约束）？
2. 如何在不改变MCP工具签名的前提下实现并发控制？
3. 如何处理Wrapper L477-480的重新加载竞态？
4. 是否需要数据库？如果需要，选Postgres还是SQLite？

### 问题2: SOP调度器幂等性

**现状**：依赖rules.yaml状态条件 + scheduler_goal_mapping为空数组

**可能方案**：
- 方案A：补全scheduler_goal_mapping，触发时立即标记
- 方案B：在wrapper.py L153添加goal级别幂等检查
- 方案C：引入分布式锁保护规则检查
- 方案D：将pending_goal字段用起来（作为幂等标志）

**请思考**：
1. 最小化改动的方案是什么？
2. 如何确保多worker场景的幂等性？
3. 是否需要修改SOP配置文件（注意：不建议）？
4. 如果Agent未调用update_sop_state，如何兜底？

### 问题3: BM25冷启动优化

**现状**：多worker重复加载JSONL + jieba分词（150ms×N）

**可能方案**：
- 方案A：进程内锁（asyncio.Lock）
- 方案B：Redis分布式锁
- 方案C：Redis缓存序列化的BM25索引
- 方案D：预加载（启动时warm up）
- 方案E：接受现状（影响范围小）

**请思考**：
1. 是否值得优化（考虑影响范围：仅冷启动前10-20个请求）？
2. 如果优化，需要引入Redis吗？
3. 序列化BM25索引的可行性（pickle? 兼容性？）

### 问题4: 存储层选型

**现状**：JSON文件（data/sessions/**/*.json）

**可能方案**：
- 方案A：保持文件 + 添加锁
- 方案B：SQLite（单文件数据库）
- 方案C：Postgres（分布式数据库）
- 方案D：Redis（纯内存 + AOF持久化）
- 方案E：混合方案（Postgres持久化 + Redis缓存）

**请思考**：
1. 是否需要数据库（考虑当前并发规模：100-120用户）？
2. 如果引入数据库，数据迁移的风险如何控制？
3. 如何实现降级策略（数据库故障时切回文件）？
4. 文件锁（fcntl.flock）能否满足需求？
